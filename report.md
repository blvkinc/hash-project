Hash Monitor: LLM-Assisted File Integrity Monitoring with Intelligent Notification Prioritisation

ABSTRACT

Context
File integrity monitoring (FIM) is a well-established technique for detecting unauthorised modifications to critical system files by comparing cryptographic hashes against a known-good baseline. However, modern computing environments are characterised by continuous updates, automated patching, and frequent configuration changes across multiple operating systems. This creates a severe signal-to-noise problem: traditional FIM tools report every hash mismatch equally, generating a flood of notifications that overwhelms users and buries genuinely suspicious changes among routine, expected modifications. Research consistently shows that alert fatigue is one of the most significant barriers to effective security monitoring, with studies indicating that over 90% of security alerts are false positives or low-priority events (Vielberth et al., 2020).

Aim
This project aims to design, build, and evaluate a cross-platform hash-based file integrity monitor that integrates a locally hosted Large Language Model (LLM), served via Ollama, to intelligently analyse, classify, and prioritise detected file system changes. The central focus is on the notification prioritisation pipeline: ensuring that the user is only alerted when something genuinely warrants attention, rather than being bombarded with every detected change. The system will operate across Linux, Windows, and macOS.

Method
A design science methodology will be used to iteratively develop and evaluate the artefact. The hash monitor will be developed in Python with cross-platform support, using SHA-256 hashing and a SQLite database. Detected changes will be fed through a multi-stage prioritisation pipeline: first a rule-based pre-filter classifies files by a tiered criticality model derived from established FIM research (Tripwire, OSSEC, MITRE ATT&CK), then the Ollama-hosted LLM performs contextual analysis to assign a final severity score. Notifications are dispatched through a priority queue with configurable thresholds, batching, suppression, and escalation logic. The system will be evaluated against controlled scenarios measuring classification accuracy and notification reduction ratio.

Anticipated Results
The project is expected to demonstrate that combining a tiered file-criticality model with LLM-assisted contextual analysis can reduce notification volume by an order of magnitude without missing critical security events. The resulting artefact will be a practical, open-source tool suitable for individuals, small organisations, and educational environments. Findings will contribute to the emerging research area of applying generative AI to security operations, specifically the under-explored intersection of LLMs and file integrity monitoring.

1. INTRODUCTION
The integrity of files on a computing system is fundamental to its security. Unauthorised modifications to system binaries, configuration files, or sensitive data can indicate a security breach, malware infection, or insider threat. File integrity monitoring addresses this by computing cryptographic hashes of files at a known-good baseline and periodically re-hashing to detect changes (Kim and Spafford, 1994). Tools such as Tripwire, OSSEC, and AIDE have long served this purpose.

However, the fundamental problem with traditional FIM is not detection but notification. A single operating system update on Linux can modify hundreds of files under /usr/bin and /lib simultaneously. A Windows Update cycle touches thousands of files across System32, SysWOW64, and the WinSxS store. On macOS, system patches alter files protected by System Integrity Protection. In each case, every modified file represents a hash mismatch that a naive monitor would report to the user. The result is alert fatigue: users become desensitised to notifications and begin ignoring them entirely, which is precisely the condition that attackers exploit (Atzeni and Lioy, 2006). A Trend Micro survey found that 51% of SOC teams feel overwhelmed by alert volume, with analysts spending over 25% of their time on false positives (Tariq et al., 2025).
This project addresses the notification problem directly. Rather than treating all hash mismatches equally, the proposed system implements a multi-stage prioritisation pipeline that combines a static, research-backed file-criticality model with dynamic, context-aware analysis from a locally hosted LLM served through Ollama. The goal is to ensure that when the user receives a notification, it genuinely warrants their attention. Changes that are expected, routine, or low-risk are silently logged or batched into periodic digests, while changes that are genuinely suspicious trigger immediate alerts with clear, LLM-generated explanations of why the change is concerning.

The key research question is: can a multi-stage prioritisation pipeline combining file-criticality tiers with LLM-assisted contextual analysis reduce notification volume in file integrity monitoring without compromising the detection of genuinely malicious changes? A secondary question is whether such a system can operate effectively across Linux, Windows, and macOS using a single unified architecture.

2. BACKGROUND
File integrity monitoring has its roots in early intrusion detection research. Tripwire, introduced by Kim and Spafford (1994), established the foundational approach of hashing files against a secure baseline. Its policy system assigns severity levels to different file categories: system binaries at severity 100 (highest), configuration files at medium severity, and log files at low severity (Open Source Tripwire, 2018). OSSEC and Wazuh extended this with real-time monitoring and cross-platform support across Windows, Linux, and macOS (Bray, Cid and Hay, 2008). Microsoft Defender for Cloud provides FIM that monitors both files and Windows Registry keys, with built-in recommended paths based on known attack patterns (Microsoft, 2024). These tools demonstrate that the field has well-established file-criticality hierarchies, but they rely predominantly on static rules for notification management.

2.1 The notification problem in file integrity monitoring
The core challenge this project addresses is not detection but notification management. PCI DSS 4.0 Requirement 11.5.2 mandates that FIM tools alert personnel to unauthorised modifications of critical files, and Requirement 10.3.4 specifies that log integrity must be monitored (PCI SSC, 2022). However, as Qualys (2023) notes, the standard explicitly recognises that monitoring files which change frequently generates noise, and recommends that FIM solutions focus on files that do not regularly change. This regulatory guidance implicitly acknowledges the notification problem without solving it.
Alert fatigue in security operations is extensively documented. Tariq et al. (2025), in their ACM Computing Surveys paper on alert fatigue in SOCs, categorise mitigation approaches into automation, augmentation, and collaboration. They identify alert prioritisation and false alert reduction as the two primary automation strategies, noting that ML-based alert management and context-based alert ranking are the most promising directions. Zhao et al. (2024) introduce AlertPro, a reinforcement-learning framework for alert prioritisation that uses context features from alert sequences and historical analyst feedback. Their work demonstrates that context-aware prioritisation significantly outperforms static severity scoring. However, both studies focus on SIEM-level alerts rather than the specific characteristics of FIM notifications, where the challenge is distinguishing legitimate system maintenance from adversarial activity.

2.2 File criticality tiers and cross-platform considerations
Research and industry practice converge on a tiered model of file criticality. Tripwire’s policy system uses severity scores from 33 (low) to 100 (critical). The MITRE ATT&CK framework documents specific persistence techniques that involve modifying system files: T1554 (Compromise Host Software Binary), T1543 (Create or Modify System Process), and T1546 (Event Triggered Execution) (MITRE, 2024). These techniques map directly to specific file paths on each operating system. Table 1 presents the proposed four-tier criticality model derived from this literature, showing how the same logical category maps to different paths across operating systems.
On Windows, the Registry presents a unique monitoring challenge. Persistence mechanisms such as Run/RunOnce keys (HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run) and service configurations (HKLM\SYSTEM\CurrentControlSet\Services) must be monitored alongside files (Microsoft, 2024). On macOS, LaunchDaemons and LaunchAgents stored as plist files under /Library/LaunchDaemons/ and /Library/LaunchAgents/ serve an equivalent persistence role (MITRE, 2024). The proposed system abstracts these OS-specific paths behind a unified criticality tier, allowing the prioritisation logic to operate identically regardless of platform.

2.3 LLMs for security alert triage
Large language models have shown increasing promise in cybersecurity applications. Thapa et al. (2022) demonstrated transformer-based models for vulnerability detection, while recent work has explored LLMs for threat intelligence summarisation and security log interpretation. The specific application of LLMs to FIM alert triage remains largely unexplored, representing a clear research gap. Unlike rule-based approaches, LLMs can reason about context: a change to /usr/bin/sudo outside of a package manager transaction is qualitatively different from the same change during an apt upgrade, and an LLM can articulate this distinction.
Ollama provides a practical framework for locally hosting open-weight models such as Llama, Mistral, and Phi. Local hosting is essential for this application because file paths, hash values, and system metadata constitute sensitive information that should not leave the monitored host. The LLM’s role in the proposed system is specifically as a second-stage classifier: it receives changes that have already passed through the static tier-based pre-filter, and its task is to assess whether the change is likely benign, suspicious, or clearly malicious, along with a confidence score and a human-readable explanation that forms the notification content.


A critical design consideration is prompt engineering for classification consistency. The system prompt will include few-shot examples of benign changes (e.g., a binary changing after apt upgrade with matching dpkg log entries) and suspicious changes (e.g., a system binary modified at 3am with no corresponding package manager activity). The LLM’s structured JSON output (severity, confidence, explanation) enables programmatic integration with the notification pipeline.


3. METHOD
The project follows a design science research methodology (Hevner et al., 2004), with the primary artefact being a cross-platform hash monitor with an LLM-assisted notification prioritisation pipeline. Development proceeds in four phases, with the notification system as the central contribution.

Phase 1: Cross-Platform Hash Engine. A Python-based monitoring tool will hash files using SHA-256 and store filename-hash pairs with metadata (timestamps, permissions, ownership) in a SQLite database. The Python watchdog library will provide cross-platform filesystem event monitoring, abstracting inotify (Linux), FSEvents (macOS), and ReadDirectoryChangesW (Windows). On Windows, the winreg module will additionally monitor Registry key changes. Each OS will ship with a default monitored-paths configuration derived from the tiered criticality model in Table 1. Phase 2: Notification Prioritisation Pipeline. This is the core contribution. The pipeline operates in three stages. Stage A is a Static Pre-Filter using the file-criticality tiers. Every detected change is immediately classified into Tier 1 through 4 based on its file path and OS-specific rules. Tier 4 changes (temp files, caches, log appends) are silently logged and never reach the LLM, eliminating the bulk of noise at near-zero computational cost. Tier 1 changes (system binaries, authentication files) bypass the LLM and trigger an immediate alert, as these should essentially never change outside tracked updates. Stage B is the LLM Contextual Analysis. Tier 2 and 3 changes, which represent the ambiguous middle ground, are passed to the Ollama-hosted LLM with a structured prompt containing: the file path, tier classification, file type and permissions, previous and current hash, modification timestamp, and critically, correlated system context. On Linux, this context includes recent entries from /var/log/dpkg.log or /var/log/yum.log. On Windows, it includes recent Windows Update activity and MSI installer logs. On macOS, it includes /var/log/install.log and Homebrew logs. The LLM returns a JSON response with a severity score (1–10), a confidence value, and a human-readable explanation. Stage C is the Notification Dispatch Engine. This stage implements the actual notification logic based on the outputs of stages A and B. The rules are: severity 8–10 triggers an immediate desktop/email notification with the LLM explanation; severity 4–7 is accumulated and dispatched as a batched digest at configurable intervals (e.g., hourly or daily); severity 1–3 is silently logged for audit purposes. Additionally, a pattern detection module monitors for accumulation effects: if multiple medium-severity changes occur in a short window, or if changes follow a known attack pattern (e.g., binary modification followed by new cron job), the system escalates the batch to an immediate alert. Phase 3: Cross-Platform Testing and Evaluation. Controlled test environments on Linux (Ubuntu), Windows 10/11, and macOS will be configured with baselines. Test scenarios include: (a) legitimate system updates (apt upgrade, Windows Update, macOS softwareupdate) to measure false positive suppression; (b) manual configuration edits to test Tier 2/3 classification; (c) simulated adversarial modifications (replacing a system binary, adding a persistence mechanism, modifying authentication files) to measure detection sensitivity; and (d) mixed scenarios combining legitimate and malicious changes. Metrics include: Notification Reduction Ratio (alerts generated vs. total changes detected), True Positive Rate for critical changes, False Negative Rate, Classification Consistency across repeated LLM calls, and user-perceived notification quality via a small user study. Phase 4: Evaluation of LLM Model Suitability. Different Ollama-hosted models (Llama 3, Mistral, Phi-3) will be compared for classification accuracy, response latency, and resource consumption to determine the minimum viable model size for effective prioritisation. Key risks include LLM classification inconsistency, which will be mitigated through prompt engineering with few-shot examples and temperature control. If local models prove insufficiently capable, the architecture supports substitution of any Ollama-compatible model. Cross-platform filesystem differences will be abstracted through a platform-specific adapter layer with a unified interface.


4. CONCLUSION
This project addresses the most significant practical barrier to effective file integrity monitoring: notification overload. By combining a research-backed, cross-platform file-criticality tier model with LLM-assisted contextual analysis, the proposed system aims to transform raw hash mismatches into prioritised, explainable notifications that respect the user’s attention. The multi-stage pipeline design ensures that the vast majority of benign changes are filtered before reaching the LLM, making the system both computationally efficient and practically useful. The artefact will be a functional, evaluated, cross-platform hash monitor that demonstrates whether locally hosted LLMs can meaningfully improve the signal-to-noise ratio in file change detection. The work contributes to the CyBOK knowledge areas of Applied Cryptography, Authentication and Accountability, Forensics, and Software Security, while also advancing the emerging field of LLM-assisted security operations. The tool is designed to be accessible to individuals and small organisations who lack dedicated SOC infrastructure but need practical file integrity monitoring that does not drown them in alerts.
